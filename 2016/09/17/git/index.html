Git

Miscellaneous

Git working environment：
Workspace -> Staging -> Remote

  • HEAD：The most recent commit on the current branch.
  • ORIG_HEAD: The previous version of HEAD.
  • FETCH_HEAD: The head of the last branch fetched and is valid only after a fetch operation.
  • MERGE_HEAD:The tip of the other branch when in the progress of a merge.

Merge

Fast-forward Merge

  1. Master branch is at commit 2.
  2. Check out a feature branch and proceed to commit 4. At the same time, no progress on master.
  3. At master branch, do git merge, will do a fast-forward merge: make master point to the head of feature branch.

Three-way Merge

  1. Master branch is at commit 2.
  2. Checkout a feature branch and proceed to commit 4. At the same time, master proceeds to commit 6.
  3. At master branch, do git merge, cannot do a fast-forward merge in this case, need to do a 3-way merge. “Three” refers to the three red commits (the common ancestor of master and feature branches, the last commit of feature branch and the last commit of master branch). 3-way merge will create a “merge commit” and make all changes happend between 2 and 4、6 in commit 7.

Rebase

In development, we have the scenario which needs to do a three-way merge often. Rebase gives another choice in this case and will make the “log graph” more beautiful.

Instead of doing git merge <feature branch> on master, do git rebase on feature branch. git rebase will:

  1. Reset feature branch’s head to the last commit of master.
  2. Reapply every change made in master but not in feature branch again (in this case, will re-commit 3 and 4 as 3’ and 4’). Note: since git rebase is “re-committing” so it will generate new commits for all commits in this branch.
  3. Now the commit graph becomes the third stage in the picture above, so doing a merge in master will be a fast-forward merge.

For more usage about rebase, please refer to : https://blog.yorkxin.org/2011/07/29/git-rebase

Diff

  • git diff: Difference between urrent workspace and staging.

  • git diff <commit>: Difference between current workspace and this commit.

    Common usgae: git diff HEAD difference between current workspace and current branch’s last commit

  • git diff --cached <commit>: Difference between current staging and this commit.

    Common usage: git diff --cached HEAD or just git diff --cached difference between current staging and current branch’s last commit.

  • git diff <commit1> <commit2>: Difference between commit1 and commit2.

Detached Head

  1. HEAD is pointing to the latest commit of master branch (commit 4).
  2. Do git checkout <commit2> will make HEAD point to commit 2, which has no branch attached to it, so this is called “You are in a ‘detached head’”. You can treat this branch as an “anonymous” branch.
  3. If commit in this status, HEAD will continue pointing to the new commit 5.
  4. Do git checkout <master> or git checkout <some branch> to leave this detached head status. Once you left, there is no way to get back to it:( But, if you do git checkout -b <new branch> in the detached head status, there will be a new branch attached to it, which can be used for getting back :)

Reset

General usage：git reset [--hard|soft|mixed|merge|keep] <commit> will reset current branch to certain commit and retain or discard changes in workspace and staging environment according to options.

  • --hard: Reset workspace and staging, discard all changes made after commit, point HEAD to commit.
  • --soft: Nothing happens for workspace and staging. Only point HEAD to commit. All changes made after commit will be in staging.
  • --mixed(default mode): Only reset staging but not workspace.

Common usage:
After making changes in the workspace or staging, you want to discard all changes (both in workspace and staging), do git reset HEAD --hard :)
